本設計書は、お墓探しナビにおける

データガバナンス設計を定義する。

目標：

1. データ破壊ゼロ
2. 不整合ゼロ
3. 重複ゼロ
4. 誤掲載ゼロ
5. 意図しない削除ゼロ

## 1. データ分類定義

### 1.1 マスターデータ

- 都道府県
- 市区町村
- 自治体コード
- municipalitySlug

特性：変更頻度低、厳密整合必須、削除禁止

### 1.2 施設データ

- 寺院
- 霊園
- 墓地
- 納骨堂

特性：更新あり、削除原則禁止（非公開化対応）

### 1.3 トランザクションデータ

- 問い合わせ
- 資料請求

特性：履歴保存必須、削除禁止

### 1.4 メタデータ

- slug
- version
- createdAt / updatedAt / publishedAt
- linkStatus / linkType

---

## 2. 削除ポリシー

絶対禁止：
- 物理削除（DELETE）

原則：
- 論理削除（isPublished = false）
- status = INACTIVE

問い合わせデータ：
削除禁止（法的保全）

---

## 3. バージョン管理設計

全更新対象テーブルに：

- version カラム必須
- 楽観ロック適用
- 更新履歴ログ保存

将来拡張：AuditLogテーブル導入可能

---

## 4. データ入力制御

### 4.1 管理画面制御

- 必須項目バリデーション
- 文字数制限
- 不正文字排除
- URL形式検証
- PDF_ONLYルール強制

### 4.2 PDF管理ルール

- PDF直リンクは url に入れない
- pdfUrl に格納
- PDF_ONLY の場合：
  - url = null
  - pdfUrl = 必須
  - linkStatus = PDF_ONLY
  - linkType = PDF

違反時は保存拒否

---

## 5. 重複防止設計

対象：
- 施設名
- slug
- 自治体コード

対策：
- DBユニーク制約
- slug一意制約
- 正規化比較（全角半角吸収）

---

## 6. データ整合性監査

月次監査項目：

- orphanレコード
- FK不整合
- municipalitySlug不一致
- null許容誤設定
- broken画像URL
- broken pdfUrl

監査結果はログ保存

## 7. 変更フロー管理

3. テスト
4. Preview確認
5. 本番反映

禁止：本番DB直接編集
