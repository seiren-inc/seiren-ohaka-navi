## 0. 目的

本設計書は、
お墓探しナビを「従来型キーワード検索」から
「AI時代に最適化されたインテリジェント検索基盤」へ進化させるための設計を定義する。

目標：

1. 検索精度最大化
2. ユーザー意図理解
3. 検索離脱率低下
4. LLM時代への適応
5. SEOとAI検索の両立

---

## 1. 現状検索構造

現在想定：
- 都道府県 → 市区町村 → 施設一覧
- slugベースルーティング
- DBクエリによる一覧表示
- フィルタ機能（将来拡張）

課題：
- キーワード一致のみ
- 意図推定なし
- 同義語対応なし
- 比較検索が弱い

## 3. 正規化・同義語設計

### 3.1 正規化ルール

- 全角半角統一
- カタカナ/ひらがな吸収
- 空白除去
- 記号除去

### 3.2 同義語辞書

| 正規語 | 同義語 |
|--------|--------|
| 霊園 | 墓地 |
| 樹木葬 | 自然葬 |
| 永代供養 | 合祀 |
| 納骨堂 | 屋内墓 |

実装方針：
- Synonymテーブル作成
- 検索時に展開

## 4. スコアリング設計

ランキング要素：

| 要素 | 重み | 備考 |
|------|------|------|
| 地域一致度 | 50 | |
| キーワード一致数 | 20 | |
| 情報充実度 | 10 | |
| 有料加点 | 15 | Doc-10連動 |
| CTR | 5 | 将来 |

```
score = 地域一致 * 50 + キーワード一致 * 20 + 情報充実度 * 10 + 有料加点 * 15 + CTR * 5
```

## 6. セマンティック検索設計（Phase 3）

目的：
「安い横浜の樹木葬」「駅近でバリアフリーの納骨堂」のような自然文検索に対応

### 6.1 Embedding対象

- 施設説明文
- 記事コンテンツ
- FAQ

### 6.2 構成案

- 施設説明をEmbedding生成
- Vector DB保存（将来：pgvector）
- 類似度検索

## 7. AI推薦設計

例：
- 「この地域で人気の施設」
- 「価格帯が近い他施設」
- 「樹木葬を探している方はこちら」

原則：
- 過度な自動推薦禁止
- 有料上書き禁止（スコア内加点のみ）

---

## 8. AIとSEOの両立

禁止：
- JS依存検索のみ
- 検索結果がindex対象になること
- 重複動的URL生成

必須：
- 主要ページは静的生成
- AI検索はUX補助
- canonical維持
