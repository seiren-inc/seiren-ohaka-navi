## 1. テスト設計方針

本プロジェクトは以下を最優先とする：

1. 保存失敗ゼロ
2. 破壊的変更ゼロ
3. 管理画面事故ゼロ
4. SEO崩壊ゼロ

テストは「機能確認」だけでなく、「事故予防」を目的とする。

---

## 2. テスト分類

### 2.1 単体テスト（Unit Test）

対象：
- APIロジック
- 保存処理
- version競合
- バリデーション

確認項目：
- 正常保存できるか
- version競合時に409を返すか
- 不正入力で400を返すか

### 2.2 結合テスト（Integration Test）

対象：
- API → Prisma → Supabase
- 管理画面CRUD
- 画像アップロード

確認項目：
- データがDBに正しく保存される
- 更新後にversionが+1される
- Storage URLが正しく保存される

### 2.3 E2Eテスト（UI確認）

対象：
- 検索 → 詳細 → 問い合わせ
- 管理画面ログイン → 編集 → 保存
- 404ページ
- エラーハンドリング

確認項目：
- 二重送信防止が機能する
- 保存後に完了画面が表示される
- エラー時に適切な表示

---

## 3. 保存保証テスト

### 3.1 楽観ロックテスト

手順：
1. 同一レコードを2画面で開く
2. 片方を更新
3. もう片方を更新

期待値：
- 2回目は409返却

### 3.2 二重送信テスト

手順：
1. フォーム送信
2. 連打

期待値：
- 1件のみ保存

## 4. セキュリティテスト

- 未ログインでadminアクセス → 401
- allowlist外ユーザー → 403
- SQLインジェクション文字列入力 → 正常拒否
- XSS文字列入力 → エスケープ
